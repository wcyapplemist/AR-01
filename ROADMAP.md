# AR Spatial Positioning — ROADMAP

## 项目目标

一个手机 Web AR 应用。用户通过手机摄像头扫描现实空间中放置的 QR Code，
在屏幕上叠加显示 3D 模型，并实时追踪手机自身在空间中的位置和姿态。

### 核心功能

1. **Start 按钮**：用户按下 Start 后，手机当前空间坐标定义为 (0,0,0)，
   此后所有运动都以该点为原点计算
2. **QR Code 空间参照**：QR Code 中硬编码了一个"真实坐标"。
   手机扫描到 QR Code 的瞬间，通过计算相对空间距离得到自己的"真实坐标"。
   QR Code 仅用于提供绝对坐标参照，空间中可放置多个不同位置的 QR Code
3. **传感器运动追踪**：手机在空间中运动引起的相对坐标变化，
   使用手机内部传感器（加速度计、陀螺仪等）计算，
   不依赖持续视觉追踪
4. **姿态追踪**：手机的旋转、倾斜角度等空间变化也需获取，
   相对于 Start 瞬间的所有 6DoF 变化都应被考虑
5. **漂移校正**：当 QR Code（作为 AR marker）在摄像头视野内时，
   使用视觉追踪校正传感器累积漂移（仅影响绝对坐标，不影响相对坐标）
6. **3D 渲染**：在摄像头画面上叠加渲染 3D 模型
7. **HUD 实时显示**：屏幕上实时显示位置、距离、角度等空间数据

## 技术栈

| 项目        | 选择                                  |
| ----------- | ------------------------------------- |
| 语言        | TypeScript                            |
| 框架        | Next.js (App Router)                  |
| 3D 渲染     | Three.js（不使用 A-Frame）            |
| AR 视觉追踪 | AR.js Three.js module（辅助漂移校正） |
| 传感器定位  | DeviceMotion + DeviceOrientation API  |
| QR 解码     | jsQR / BarcodeDetector API            |
| 样式        | Tailwind CSS                          |
| 后端        | Next.js API Routes + JSON 文件存储    |

## 核心架构

```
[Start Button] → 设原点 (0,0,0) → 启动传感器追踪
       ↓
[手机传感器] → 持续追踪相对位移 + 旋转 (6DoF)
       ↓
[QR Code 扫描] → 提取硬编码真实坐标 → 计算手机绝对位置
       ↓ (QR 在视野内时)
[AR.js 视觉追踪] → 校正传感器漂移 (仅绝对坐标)
       ↓
[Three.js 渲染] → 3D 模型叠加 + HUD 数据显示
```

## 项目结构要求

项目应按职责划分为三个独立的模块区域，各自可独立运行和演示：

| 模块区域              | 职责                                                          | 依赖                     |
| --------------------- | ------------------------------------------------------------- | ------------------------ |
| **AR 模块**           | 摄像头 + 传感器追踪 + AR.js 漂移校正 + 空间定位 + 3D 叠加渲染 | Three.js, AR.js, 传感器 API |
| **QR Code 模块**      | 摄像头 + jsQR 解码 + QR 内容验证与解析 + 结果展示            | jsQR                     |
| **Three.js 模块**     | 纯 3D 场景渲染（Scene, Camera, 灯光, 模型, 动画）            | Three.js                 |

- 每个模块区域可独立运行（有各自的页面路由），也可组合为完整的集成页面
- 模块间通过共享类型定义通信，避免直接相互依赖
- 模块内使用相对导入，跨模块引用使用路径别名

## 双坐标系设计

项目使用两套独立的坐标系，同时维护：

### 相对坐标 (Relative)

- **含义**：手机相对于 Start 瞬间原点 (0,0,0) 的位移
- **数据来源**：纯手机传感器积分（加速度计 + 陀螺仪）
- **特点**：持续累积，不受漂移校正影响
- **用途**：反映手机从起点开始的总运动量

### 绝对坐标 (Absolute)

- **含义**：手机在真实世界中的位置
- **数据来源**：QR Code 校准 + AR.js 视觉校正
- **计算方式**：扫描 QR 时，`绝对坐标 = QR真实坐标 + 手机相对QR的偏移`
- **特点**：受漂移校正影响，精度更高
- **用途**：定位手机在物理空间中的真实位置

### 漂移校正原则

- 漂移校正**仅修正绝对坐标**，不回改传感器原始数据
- 相对坐标永远从 (0,0,0) 开始持续累加，保持传感器数据的完整性

## QR Code 设计

- QR Code 内容为 JSON 格式：`{"id":"qr-001","pos":{"x":5.0,"y":3.0,"z":0.0}}`
- `id`：唯一标识符
- `pos`：该 QR Code 在真实世界中的绝对坐标 (x, y, z)，单位米
- 空间中可放置多个 QR Code，各自携带不同的真实坐标
- QR Code 同时可作为 AR.js 的 pattern marker，用于视觉追踪漂移校正

## 关键技术约束

1. **惯性导航漂移**：加速度计双重积分会快速累积误差。
   这是传感器定位的固有问题，需要视觉校正补偿
2. **传感器权限**：iOS Safari 需显式调用
   `DeviceMotionEvent.requestPermission()`，Android 行为因版本而异
3. **SSR 兼容**：所有传感器/AR/3D 组件需客户端加载，
   不可在服务端渲染
4. **移动端优先**：应用面向手机使用，需在 iOS Safari 和
   Android Chrome 上测试验证
5. **坐标系对齐**：传感器坐标系需与"世界坐标系"对齐，
   初始方向校准是关键步骤
